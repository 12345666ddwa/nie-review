# 从PPT+录音到交互式复习网页的完整工作流

## 适用场景

用户提供课程复习PPT和/或课堂录音转写文本，需要生成一个覆盖全部考点的交互式复习网页，可部署到公网分享给同学。

## 总流程

```
PPT → python-pptx提取 → 考点清单
                              ↓
录音转写/笔记 ──→ 交叉对比(查漏补缺)
                              ↓
                    内容架构(教学逻辑)
                              ↓
                    HTML网页构建(单文件)
                              ↓
              美工 → 自测系统 → 署名 → 部署
```

## 阶段一：输入提取

### 1. PPT提取规则

```python
from pptx import Presentation
prs = Presentation("复习课.pptx")
for i, slide in enumerate(prs.slides, 1):
    for shape in slide.shapes:
        if shape.has_text_frame:
            for para in shape.text_frame.paragraphs:
                # 保留 para.level（缩进层级）区分主/子知识点
                text = para.text.strip()
        if shape.has_table:
            # 遍历 table.rows → row.cells → cell.text
```

**铁律：**
- 遍历每一页的每一个shape（文本框+表格），不能只看标题
- 保留层级信息（para.level），区分主知识点和子知识点
- PPT列出的知识点就是考试范围，不能自行删减

### 2. 录音转写/笔记处理规则

- 课堂笔记是PPT的"扩充"而非"替代"
- 转写中的学术人名可能有听音误（如"洛斯"→"沃利斯和诺思"），需交叉核实
- 笔记中超出PPT范围的内容作为加分扩充保留，不丢弃

## 阶段二：交叉对比

### 程序化验证（必做）

```python
checklist = {
    "第一章 标题": [
        ("知识点名", "搜索关键词"),
        ...
    ],
    ...
}
for chapter, items in checklist.items():
    for name, keyword in items:
        if keyword in html_content:
            print(f"✅ {name}")
        else:
            print(f"❌ {name}")
```

**规则：**
- 逐条对照：PPT上的每一个关键词都必须在最终内容中找到对应
- 三类差异处理：
  - PPT有笔记没有 → 补充说明
  - 笔记有PPT没有 → 作为扩充保留
  - 术语冲突 → 以教材/学术标准为准
- 内容写完后再跑一次全量检测脚本，确保无遗漏
- 目标：100%覆盖率

## 阶段三：内容组织

### 每个知识点必须包含的四要素

1. **定义/内容**：先直觉理解再学术定义（假设读者零基础）
2. **题型预测标签**：根据特征匹配（分类→多选、定义→名词解释、是非→判断、逻辑链→论述）
3. **易错陷阱**（trap-box）：指出常见错误理解
4. **自测练习题**（self-test）：可折叠的练习+答案

### 题型预测规则

| 知识点特征 | 适配题型 |
|-----------|---------|
| 多维分类/多因素 | 多选 |
| 概念定义 | 名词解释、单选 |
| 容易混淆的表述 | 判断 |
| 理论体系/逻辑链 | 论述 |
| 图形+填空 | 综合分析 |

### 自测系统设计规则

- **两层折叠**：第一层展开看题目，第二层揭晓答案
- **题目用考卷格式**：标注【判断题】【单选题】等，选项ABCD
- **答案包含解析**：说明为什么对/错，指出陷阱
- **分布均匀**：每个章节的每个主要知识模块都必须有自测
- **覆盖所有题型**：单选/多选/判断/名词解释都要出现

### 论述题标答格式（老师要求小论文）

```
第一步：定义核心概念（2行）
第二步：理论观点梳理（4-5行）
第三步：个人理解与评价（3行）
第四步：结合现实案例（3-4行）
```

### 综合分析题规则

- 可能含图形（SVG内嵌，如供求曲线）
- 10个填空，有长有短（单字到一句话）
- 前半部分微观基础（均衡点、出清等），后半部分结合课程理论
- 答案同样折叠可揭晓

## 阶段四：HTML构建

### 单文件架构

所有CSS/JS/SVG内联，零外部依赖。结构：

```html
<!DOCTYPE html>
<html>
<head>
  <style>/* 全部CSS */</style>
</head>
<body>
  <nav class="top-nav">顶部导航</nav>
  <div class="main-content">
    署名 + 学校信息
    考试概览
    第一章～第N章（知识点+自测+陷阱）
    应试策略（论述标答+综合分析+图形题）
    速记清单（名词解释+判断陷阱+多选高频）
  </div>
  <script>/* 动画+导航高亮 */</script>
</body>
</html>
```

### CSS设计体系

| 组件 | 用途 | 配色 |
|------|------|------|
| `.card-blue` | 定义/重点 | 蓝色系 |
| `.card-red` | 警告/批判 | 红色系 |
| `.card-green` | 正面/答案 | 绿色系 |
| `.card-orange` | 提示/补充 | 橙色系 |
| `.card-purple` | 特殊概念 | 紫色系 |
| `.trap-box` | 易错陷阱 | 浅红底 |
| `.self-test` | 自测练习 | 紫色虚线框 |
| `.exam-question` | 模拟题 | 白底+边框 |
| `.kw` | 关键字蓝色 | 蓝色加粗+下划线 |
| `.kw-red` | 关键字红色 | 红色加粗 |
| `.key-point` | 重点句 | 黄底棕字 |

### 动效清单

- IntersectionObserver 滚动入场淡入
- 卡片悬停上浮+阴影增强
- 顶部导航当前章节高亮
- 折叠面板展开动画
- 页面标题淡入效果

### 关键字高亮规则

定义中的核心术语用 `.kw`（蓝色），易错/重要词用 `.kw-red`（红色），完整重点句用 `.key-point`（黄底）。目的：防止大段文字阅读走神。

## 阶段六：部署

### GitHub Pages 部署步骤

```bash
# 1. 创建仓库
curl -X POST https://api.github.com/user/repos \
  -H "Authorization: token $TOKEN" \
  -d '{"name":"repo-name","public":true,"auto_init":true}'

# 2. 上传（大文件用@file）
# 先base64编码HTML → 写入JSON payload → curl -d @payload.json
curl -X PUT .../contents/index.html \
  -H "Authorization: token $TOKEN" \
  -d @payload.json

# 3. 开启Pages
curl -X POST .../pages \
  -d '{"source":{"branch":"main","path":"/"}}'
```

### 部署注意事项

- Token通过临时文件传递（分段拼接写入防redact）
- base64后>100KB时不能放命令行参数，必须写入JSON文件用`-d @file`
- 更新文件需先GET获取当前SHA，再PUT时带上
- 所有GitHub API请求走代理
- Pages缓存刷新需1-2分钟

## 阶段七：嵌入AI助教（可选增强）

在网页中嵌入"选中即问AI"功能，学生边看边问：

### 交互设计
- 右下角浮动🤖按钮 + 选中文字自动弹出"问AI助教"提示
- 点击后AI自动解释选中内容，支持追问（多轮对话）
- DeepSeek API流式输出，Key存localStorage（首次输入一次）

### 🔴 系统提示词必须包含完整课程知识
不能只写"你是XX课的辅导老师"——太泛，回答与老师教的对不上。必须将全部章节知识体系压缩注入system prompt：
```
你是[学校][课程]的专业AI助教。以下是课程知识体系：
【考试信息】题型/分值/答题框架
【第1章·标题】核心概念=定义；学者归属；常考陷阱
【第N章·标题】...
【答题规则】通俗+举例+对比+题型提示+不编造
```
关键：加规则"所有回答必须与上述知识体系一致，不得编造"。

### 技术要点
- 流式输出(`stream:true`)逐字显示
- `chatHistory.slice(-6)`保留最近3轮上下文
- 401错误自动清除Key提示重设
- 手机适配：面板自动全宽

## 四轮验证方法论（完成网页后必做）

| 轮次 | 检查内容 | 方法 | 示例规模 |
|------|---------|------|---------|
| 第一轮 | 概念定义覆盖 | 逐条关键字搜索 | ~114项 |
| 第二轮 | 章节结构丰度 | 统计每节的卡片/表格/陷阱/自测/字数 | ~40节 |
| 第三轮 | 内容准确性 | 抽查定理条件/学者归属/因果方向/学者人名 | ~21项 |
| 第四轮 | 原始资料逐句对照 | 回到PPT+笔记原文逐句搜细节 | ~42条 |

**第四轮最关键**：考试答题实操细节（如"答题卡涂深""试卷可打草稿答题卷不可"）和课堂顺带提及的术语（如"林达尔均衡"）在前三轮中因为"不是核心考点"很容易遗漏。必须回到原始资料逐句扫。

## 阶段七：嵌入AI助教（选中即问）

### 功能设计

在复习网页中嵌入一个AI助教系统，学生选中看不懂的文字即可当场提问，AI基于课程知识体系回答。

### 交互流程

```
学生选中文字 → 文字上方弹出「🤖 问AI助教」按钮
    → 点击 → 右下角打开对话面板
    → AI自动基于选中内容生成通俗解释
    → 学生可继续追问（支持多轮对话）
```

### 技术架构

```html
<!-- 三个核心元素 -->
<button class="ai-fab">🤖</button>          <!-- 右下角悬浮按钮 -->
<div class="ai-sel-tip">🤖 问AI助教</div>   <!-- 选中文字后的弹出提示 -->
<div class="ai-panel">...</div>              <!-- 对话面板 -->
```

- **选中检测**：`mouseup` 事件 → `window.getSelection()` 获取选中文字 → 计算位置显示提示按钮
- **API调用**：直接从前端调用 DeepSeek API（`https://api.deepseek.com/chat/completions`），支持 stream 流式输出
- **Key存储**：用户首次输入API Key → 存入 `localStorage`（仅存本地浏览器，不上传服务器）
- **对话记忆**：保留最近6轮对话历史（`chatHistory.slice(-6)`），支持追问

### 安全设计（关键）

| 层面 | 设计 | 说明 |
|------|------|------|
| Key存储 | localStorage | 每台设备×每个浏览器独立，不跟随URL传播 |
| HTML源码 | 不含任何Key | grep "sk-" 只能找到 placeholder |
| 跨设备 | 需重新输入 | 手机/电脑/同学设备各自独立 |
| 成本控制 | 每人自己的Key | 谁用谁付费，DeepSeek极便宜(约¥0.001/次) |

### System Prompt 设计（核心）

AI助教的质量取决于System Prompt。**必须注入完整课程知识体系**，否则AI会用通用知识回答，可能与老师讲的不一致。

```
System Prompt 结构：
├── 身份定位：XX大学XX课程的专业AI助教
├── 考试信息：题型/分值/答题框架
├── 第1章知识点（压缩为要点列表）
├── 第2章知识点
├── ...（逐章列出关键概念、定义、学者归属、常见陷阱）
├── 第N章知识点
└── 答题规则：通俗/举例/对比/简洁/提示题型/不编造
```

**Prompt编写原则：**

1. **压缩但不遗漏**：每个概念用"关键词=一句话定义"格式，如`机会主义=无底线追逐自利(极致利己)`
2. **标注陷阱**：在易错点后加`（陷阱：XXX=错）`，如`（陷阱：不是"越发展费用越低"）`
3. **标注学者归属**：防止张冠李戴，如`剩余索取权(阿尔钦)≠剩余控制权(GHM)`
4. **限制回答长度**：`控制在200字以内`，防止长篇大论学生看不完
5. **强制一致性**：`所有回答必须与上述知识体系一致，不得编造`
6. **考试导向**：要求AI主动提示"这个点常考XX题型"

### CSS设计要点

- 悬浮按钮：`position:fixed; right:24px; bottom:24px;` 紫色渐变圆形
- 对话面板：`position:fixed; right:20px; bottom:86px;` 白底圆角卡片，max-height限制
- 选中提示：跟随选中文字位置出现，`position:fixed` + JS动态计算坐标
- 消息气泡：用户右对齐紫色底，AI左对齐灰色底
- 响应式：窄屏时面板全宽 `@media (max-width:500px) { left:8px; right:8px; }`

### 流式输出实现

```javascript
// 关键：使用 stream:true + ReadableStream 逐字输出
const resp = await fetch(API_URL, {
  method: 'POST',
```

## Pitfalls

1. **PPT提取不能只看标题**：很多知识点藏在正文文本框和表格里
2. **课堂人名听音误**：转写可能把"沃利斯"听成"洛斯"，必须核实
3. **自测题分布必须均匀**：用户明确指出"为什么有些有例题有些没有"——每个知识模块都必须配自测，不能只在trap-box处加题而遗漏其他节。完成后用脚本统计每个section-title下是否有self-test
4. **定义必须先于性质**：用户指出"2.3交易费用的性质为什么没有定义"——讲任何概念的"性质/功能/特征"之前，必须先给出这个概念本身的定义。这是教学逻辑的基本要求
5. **美工方向先问清**：避免做完才说"太老气"然后大改。用户偏好明亮暖色系>暗沉系
6. **导航放顶部不放左侧**：用户明确"左边有点不好看"——复习网页用顶部粘性导航条（磨砂玻璃效果+章节高亮），不用侧边栏
7. **例题必须可自测**：用户要求"以题型的样子出现，下面再来写解答"——题目展示为考卷格式，答案用两层折叠（先点开看题→再点揭晓答案），让学生先思考再对照
8. **关键字词必须高亮**：用户说"不然容易看着就走神"——核心术语蓝色(.kw)、易错词红色(.kw-red)、重点句黄底(.key-point)，不能全是黑字
9. **base64太大炸命令行**：>100KB必须写入JSON文件用`curl -d @file`
10. **GitHub更新需要SHA**：不带SHA会报409冲突
11. **bash heredoc中的SHA传递**：`<< 'PYEOF'`（单引号）会阻止shell变量展开，`$SHA`变成字面文字。解决：先把SHA写入文件(`/tmp/.sha`)，Python中`open('/tmp/.sha').read().strip()`读取
12. **"覆盖100%"要用脚本验证**：人肉检查必遗漏。用关键词搜索脚本逐条验证
13. **综合分析题可能含图形**：老师明确说会出供求曲线等图形让学生描述/填空——提前准备SVG图（坐标轴+曲线+均衡点+虚线），题目前半基础微观后半结合课程理论
14. **分享给同学时要加学校信息**：署名+学校+学院+课程+学期+功能标签+使用说明，让收到链接的同学一目了然
15. **AI助教prompt不能太泛**：用户问"给ai助教设定任务基础背景了吗"——通用的"你是XX老师"无法保证回答准确。必须把全部章节的定义/学者归属/易错点/判断陷阱压缩注入system prompt，约2000字
